# Campus Agent Console 前端设计

## 目标

构建一个暖色调的单页控制台页面，并由现有 Go 服务直接托管。页面需要在不引入独立前端构建系统的前提下，展示当前后端已经具备的核心能力。

## 范围

本次设计覆盖现有 Gin 服务暴露的一个浏览器入口：

- 根页面 `/`
- 通过 `GET /healthz` 展示服务健康状态
- 通过 `POST /api/v1/chat` 发起聊天交互
- 通过 `POST /api/v1/tasks` 创建异步任务
- 通过 `GET /api/v1/tasks?user_id=...` 获取任务列表
- 通过 `GET /api/v1/tasks/:id` 刷新任务详情

本阶段不包含：

- 登录与认证
- 多页面导航
- 前端框架与构建链
- WebSocket 或 Server-Sent Events 实时推送
- Prompt 或知识文档的可视化编辑器

## 产品形态

页面是一个单屏控制台，分为三个功能区域：

1. 顶部栏
   - Campus Agent 品牌区
   - 简短产品说明
   - 后端健康状态徽标
   - 对全部请求生效的用户 ID 输入框
2. 主对话区域
   - 用户与 Agent 的对话时间线
   - 自由输入的聊天表单
   - 展示意图、任务、知识答案和工具执行结果的结构化响应卡片
3. 右侧操作面板
   - 异步任务创建表单
   - 当前用户的任务列表
   - 当前选中任务的详情卡片

桌面端采用双栏布局，移动端折叠为纵向布局，对话区在前，操作区在后。

## 视觉方向

界面应呈现“校园事务工作台”的感觉，而不是通用 SaaS 后台。

- 主色盘：暖橙、赤陶、沙色、米白
- 对比色：深棕色文本与边框
- 点缀色：仅在健康状态和成功状态上使用少量绿色
- 背景：分层暖色渐变，加轻微纸张质感
- 字体：标题使用偏衬线风格，正文使用清晰的无衬线后备字体
- 动效：页面进入时轻微淡入，卡片悬浮时轻微抬升，运行中的任务状态点可有轻微呼吸感

页面应避免紫色主调、霓虹渐变、厚重毛玻璃和暗色优先设计。

## 信息架构

### 顶部栏

顶部栏负责提供系统上下文和状态信息：

- 产品名 `Campus Agent Console`
- 面向校园任务自动化的一句话说明
- `online` / `offline` 健康状态徽标
- 带默认值的用户 ID 输入框，便于本地演示

### 对话面板

对话面板是主入口，因为它最能体现 Agent 工作流。

每条用户消息以紧凑的暖色气泡展示。每条 Agent 响应以卡片形式展示，可包含：

- 检测出的意图
- 规划出的任务列表
- 知识问答文本
- 工具执行结果项

如果后端返回错误，应在消息流里展示内联错误卡片，而不是使用浏览器弹窗。

### 操作面板

操作面板直接暴露异步任务能力，清楚展示 Worker 链路。

分区如下：

- `创建任务`
  - 任务 ID 输入框
  - 任务名称输入框
  - 提交按钮
- `最近任务`
  - 当前用户的任务列表
  - 每一项展示任务名、状态和可用时的更新时间
- `任务详情`
  - 当前选中的任务 ID
  - 状态、结果和时间信息
  - 手动刷新按钮

## 交互设计

### 页面启动

页面首次加载时：

1. 先渲染静态页面骨架
2. 请求 `/healthz`
3. 加载当前用户的任务列表
4. 如果存在任务，则默认选择最新任务并拉取详情

如果健康检查失败，页面仍然渲染，但要标记后端离线，并禁用提交按钮，直到后续健康检查恢复或用户手动刷新成功。

### 聊天流程

当用户提交消息时：

1. 立即把用户消息追加到时间线
2. 插入一张等待中的 Agent 卡片
3. 调用 `POST /api/v1/chat`
4. 用结构化响应内容替换等待状态

如果响应中包含知识答案，则作为 Agent 卡片正文展示；如果包含任务执行结果，则按结果行渲染。

### 任务创建流程

当用户提交任务时：

1. 校验用户 ID、任务 ID、任务名称是否存在
2. 调用 `POST /api/v1/tasks`
3. 将新任务插入到任务列表顶部
4. 自动选中新任务
5. 开始短轮询任务详情，直到状态变为 `success` 或 `failed`

### 任务详情流程

当用户选中某条任务时：

1. 调用 `GET /api/v1/tasks/:id`
2. 渲染任务详情卡片
3. 如果状态是 `pending` 或 `running`，则每隔几秒继续轮询
4. 状态进入终态后停止轮询

页面同一时刻只能保留一个轮询定时器，避免切换任务后出现重复请求。

## 错误处理

- 表单输入非法：在表单内联展示错误提示，且不破坏当前页面状态
- 健康检查失败：展示离线状态，并禁用操作
- 聊天请求失败：在对话区展示错误卡片
- 任务列表或任务详情失败：在对应区域展示错误文本
- 轮询失败：连续少量失败后停止轮询，并暴露手动刷新入口

错误文案应简短、具体、可就地理解。

## 可访问性与响应式

- 使用语义化表单控件和按钮标签
- 确保暖色背景上的文字对比度可读
- 聊天表单和任务表单都支持键盘提交
- 在约 360px 及以上宽度的移动端正常可用
- 避免影响阅读和操作的过强动画

## 技术设计

### 文件布局

新增文件：

- `web/static/index.html`
- `web/static/styles.css`
- `web/static/app.js`

服务端改动：

- 注册根页面路由
- 通过 `/static` 之类的固定路径提供静态资源

## 前端运行时

仅使用浏览器原生能力：

- 使用 `fetch` 发起 API 请求
- 使用 DOM 渲染辅助函数生成卡片和列表
- 使用一个轻量内存状态对象维护用户 ID、当前任务、任务列表和轮询定时器

本阶段不引入 bundler、前端框架或包管理器。

## 后端集成假设

页面直接使用当前 API 协议：

- 聊天请求体：`user_id`、`message`
- 创建任务请求体：`id`、`user_id`、`task_name`
- 任务列表按 `user_id` 过滤
- 响应统一使用 `pkg/response` 的成功/失败包裹格式

本阶段不要求修改后端数据结构。

## 测试策略

本阶段测试保持轻量，但要求明确：

- 为 `/` 和静态资源路由补充单元测试
- 验证 `go test ./...` 仍然全部通过
- 手工打开页面并确认：
  - 健康状态徽标正常加载
  - 聊天请求会渲染响应卡片
  - 创建任务后右侧列表会更新
  - 任务详情轮询会在终态停止
  - 移动端布局保持可读

## 实现注意事项

- `app.js` 只聚焦状态和渲染，不要把大量 HTML 字符串无节制塞进逻辑里
- `styles.css` 按布局、组件、响应式覆盖三个层次组织
- DOM ID 数量尽量少，能用 `data-*` 属性表达的地方优先使用
- 保持现有 Gin 路由装配方式，不新增另一套前端启动路径

## 验收标准

满足以下条件即可视为本阶段完成：

- 访问 `/` 能看到带样式的单页控制台
- 页面由现有 Go 服务直接托管
- 页面可以调用当前后端完成聊天、任务创建、任务列表加载和任务详情刷新
- 视觉风格明确偏暖色调，且不落入通用模板化样式
- 不需要额外的前端工具链
